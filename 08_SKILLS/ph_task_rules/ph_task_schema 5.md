# `tech_team_outputs.ph_task` — Schema Documentation

## Purpose

This table is the **task feed for the hosted tool**.

Workflow:

1. A **developer** publishes a periodic task (e.g. weekly or monthly), writing the task content as **HTML** in the `html_content` column.
2. The **end user** opens the hosted tool and sees that HTML rendered.
3. The user performs the action the task instructs.
4. When done, the row records **who** completed it (`action_took_by`) and **when** (`action_took_date_time`).

So a row's lifecycle is: *created by a developer → viewed by a user → acted on → marked complete*.

## Constraints

- **Primary key:** `id` (`ph_task_pkey`). Auto-incrementing identity column.

> `project_code` is intentionally **not unique** — one project has many tasks, so the
> same code appears on multiple rows. It is indexed (non-unique) for lookup; see Indexes.

---

## How "done vs not done" is determined

There is **no explicit status flag** for completion. A task is considered **complete** when `action_took_by` (and `action_took_date_time`) are populated, and **pending** while they are `NULL`.

```sql
-- pending tasks
SELECT * FROM tech_team_outputs.ph_task WHERE action_took_by IS NULL;
```

> If this implicit approach ever becomes fragile, consider adding an explicit
> `action_status` column (`'pending'` / `'done'`, default `'pending'`). Not currently implemented.

---

## Indexes

`id` is indexed automatically (by the PK) — no manual index needed. `project_code` is **no longer** auto-indexed (its UNIQUE constraint was removed), so it now has an explicit non-unique index below.

Manually created indexes:

| Index | Column(s) | Why |
|---|---|---|
| `idx_ph_task_project_code` | `project_code` | Fetching all tasks for a project (replaces dropped UNIQUE index) |
| `idx_ph_task_assigned_user` | `assigned_user` | Users loading the tasks assigned to them |
| `idx_ph_task_developer` | `developer` | Developer viewing tasks they pushed |
| `idx_ph_task_created_at` | `created_at DESC` | Newest-first task feed |
| `idx_ph_task_status` | `version_status` | Filtering by status (see caveat) |
| `idx_ph_task_task_name` | `task_name` | Exact / prefix lookup by task name |

### Index caveats for the next developer

- **`idx_ph_task_status`** is on a low-cardinality column (only a few status values). Btree indexes there often go unused; Postgres may scan instead. Keep it only if you frequently filter for a **rare** status value. Otherwise it's safe to drop.
- **`idx_ph_task_task_name`** only helps exact matches (`task_name = 'X'`) and prefix matches (`LIKE 'X%'`). It does **not** help substring search (`LIKE '%X%'`) or case-insensitive search — those need a `pg_trgm` GIN index instead.
- Indexes don't help until the table has meaningful volume. On a small table Postgres prefers a sequential scan because it's faster. Validate real usage later with `EXPLAIN ANALYZE` and check `pg_stat_user_indexes` to prune indexes that are never used.

## Design notes / gotchas

- **`assigned_user`, not `user`.** `user` is a reserved keyword in PostgreSQL. It was deliberately renamed to `assigned_user` to avoid having to double-quote it (`"user"`) in every query forever.
- **Identity column, not `SERIAL`.** `id` uses `GENERATED BY DEFAULT AS IDENTITY` (the SQL-standard approach, recommended over `SERIAL` since PG 10). `BY DEFAULT` means you *can* insert an explicit `id` when needed (e.g. data seeding); switch to `GENERATED ALWAYS` if you want to forbid that.
- **NULL and UNIQUE.** Under Postgres defaults, `NULL` is never equal to another `NULL`, so a `UNIQUE` column still allows multiple NULL rows. If you add future unique columns that must identify a row, make them `NOT NULL`. (PG 15+ offers `UNIQUE NULLS NOT DISTINCT` if you want to allow at most one NULL.)
- **`updated_at` is not auto-maintained.** It defaults to `now()` on insert but nothing updates it on row changes — so it currently mirrors `created_at` unless the application sets it. Add a `BEFORE UPDATE` trigger if you want it to refresh automatically.
- **Optional text columns are nullable, not defaulted.** A `NOT NULL` text column with no default would reject any insert that omits it. Optional fields (`team`, `task_id`, `version_status`, `action_took_by`, etc.) are left nullable on purpose so "no value yet" stays distinct from an empty string.
