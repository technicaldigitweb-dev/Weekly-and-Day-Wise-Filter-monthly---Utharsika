Row grain: one row = one ASIN + one SKU (or one ASIN + blank SKU for no-mapping/Vendor-only rows). Vendor sales (public.vendor_sales, ASIN-only, no SKU) are carried on exactly one dedicated row per ASIN - never attached to a SKU-specific row, so Vendor figures are never duplicated or multiplied across an ASIN's SKUs.
Ordered Product Sales rule (v002): Sales = the original product value ordered in the selected period, for order_status IN ('Completed','Refunded'), source_name='AMAZON'. A refund in the same or a later month does not remove the original Sales from the month the order was placed. Cancelled/Canceled rows are always excluded (they never became a valid order).
Total Orders rule (v002): Total Orders = COUNT(DISTINCT order_item_info) for order_status='Completed' and source_name IN ('AMAZON','REPLACEMENT'). A Completed, zero-value REPLACEMENT row contributes one order item and zero Sales. Refunded rows contribute to Sales (see above) but not to Total Orders. Total Orders = FBM Orders + FBA Orders only - Vendor Units are never added to Total Orders.
Total Quantity rule (v002): Total Quantity = FBM Quantity + FBA Quantity + Vendor Units (public.vendor_sales.ordered_units). Vendor Orders are not available from the current database - public.vendor_sales has no order ID, purchase-order number, or other order-level key (only a surrogate row ID, which is not a valid order count and is never used as one). Vendor sales are sourced from periods (start/end date ranges, not daily rows, stored half-open [start_date, end_date)); a period's entire revenue/units is attributed to any selected date range it overlaps (corrected half-open boundary test), without proration. A period only partially overlapping a shorter selection (e.g. a monthly Vendor period compared against a daily/weekly selection) still has its full value attributed - a genuine source-data granularity limitation, not a calculation error.
Vendor period boundary fix (v002): an earlier version's overlap test double-counted a Vendor period into the month immediately following it whenever the period's end date landed exactly on the next month's start date (common, since many Vendor periods are exact calendar months). Found and corrected during v002 Phase 7 validation, cross-checked directly against PostgreSQL - no longer occurs in this version.
Methodology
  • Row grain: one ASIN + one SKU per row (or one ASIN + blank SKU for no-mapping/Vendor-only rows). Utharsika-assigned Amazon products only (resolved via public.user → public.ph_categories → public.ph_cate_products, DISTINCT applied). The table always starts from the complete assigned-ASIN master (__UAWSO_ASSIGNED_ASIN_COUNT__ ASINs) and left-joins transaction/vendor data - it never starts from transaction data.
  • Ordered Product Sales: Amazon UK only (source_name='AMAZON', market_place='UK'), status IN (Completed, Refunded), aggregated at ASIN+SKU grain and split by the fba_sales flag. Uses item_price × quantity per line, which reproduces Amazon's own "Ordered Product Sales" Business Report metric exactly (verified against ASIN B0FX2QT3B1, June 2026).
  • Total Orders: COUNT(DISTINCT order_item_info), status='Completed', source IN (AMAZON, REPLACEMENT). Total Orders = FBM Orders + FBA Orders (Vendor excluded).
  • Total Quantity: FBM Quantity + FBA Quantity (same row scope as Total Orders) + Vendor Units. Never labelled "Orders".
  • Vendor: public.vendor_sales, ordered_revenue/ordered_units, attributed once per ASIN. Vendor Orders = N/A (no valid order key in source data).
  • Data included through __UAWSO_LATEST_COMPLETED_DATE__; current-day partial data excluded.
  • Trend is Sales-based only (on CY vs PY Sales): UP / DOWN / NO CHANGE. Previous=0,Current>0 → UP. Previous=0,Current=0 → NO CHANGE.
  • Achievement % measures progress against 130% of the equivalent prior-year period. Undefined ("—") when the prior-year base is zero and current > 0 — never fabricated.
